# توثيق وحدة النظام المالي (Finance Module Documentation)

توثيق شامل للهيكل البرمجي، القواعد العملية، والواجهات البرمجية الخاصة بوحدة المالية في نظام HR API.

---

## 1. نظرة عامة (Overview)

تم تصميم وحدة المالية لإدارة الحسابات المالية للشركة، حسابات رواتب الموظفين (البنوك)، تتبع المعاملات المالية (إيداعات، مصارف، تحويلات)، وفئات القيود المالية مع الالتزام التام بعزل بيانات الشركات (Multi-tenancy) وصلاحيات الهرمية الوظيفية.

---

## 2. الهيكل البرمجي (Architecture)

يتبع النظام نمط الطبقات (Layered Architecture) لضمان فصل المسؤوليات:

- **Models**: تمثل جداول قاعدة البيانات (`FinanceAccount`, `EmployeeAccount`, `FinanceTransaction`, `ErpConstant`). تم إضافة **Accessors** لضمان ظهور مسارات المرفقات بشكل صحيح.
- **Repositories**: مسؤولة عن استعلامات قاعدة البيانات والفلترة والترقيم (`FinanceRepository`).
- **Services**: تحتوي على منطق العمل المعقد وضمان سلامة البيانات (`FinanceService`).
- **DTOs**: كائنات لنقل البيانات بين الطبقات بشكل آمن ومنظم (`CreateTransactionDTO`).
- **Requests**: طبقة التحقق (Validation). تدعم الآن معالجة الملفات المرفقة عبر `multipart/form-data`.
- **Controllers**: نقطة الدخول، تم دمج `FileUploadService` فيها لمعالجة المستندات.

---

## 3. قاعدة البيانات (Database Schema)

### المعاملات المالية (`ci_finance_transactions`)

| العمود            | النوع    | الوصف                                             |
| :---------------- | :------- | :------------------------------------------------ |
| `transaction_id`  | INT (PK) | المعرف الفريد                                     |
| `account_id`      | INT      | الحساب المتأثر                                    |
| `amount`          | DECIMAL  | قيمة المبلغ                                       |
| `dr_cr`           | ENUM     | طبيعة الحركة (dr: مصروف، cr: إيداع)               |
| `attachment_file` | VARCHAR  | **اسم الملف فقط** (يتم إضافة المجلد آلياً بالعرض) |

---

## 4. نظام المرفقات (File Upload System)

تم تطوير نظام المرفقات ليكون مرناً ومنظماً:

- **التخزين**: يتم رفع الملفات إلى مجلد `uploads/transactions/`.
- **قاعدة البيانات**: يتم تخزين اسم الملف فقط (مثل `expense_24_1770726573.png`) لضمان خفة البيانات.
- **العرض آلياً (Accessors)**: تم إضافة منطق في الموديل يقوم بإضافة المسار `transactions/` تلقائياً عند استرجاع البيانات في الـ API.
- **دعم Swagger**: تم تغيير نوع الطلب إلى `multipart/form-data` ليسمح للمطورين بتجربة رفع الملفات مباشرة من الواجهة.

---

## 5. منطق العمل والقواعد (Business Logic)

### أ. الهرمية والخصوصية (Hierarchy & Privacy)

- يتم تطبيق قاعدة **"Downstream Authority"**: لا يمكن لموظف تسجيل مصروفات لموظف أعلى منه أو في نفس مستواه.

### ب. التحقق من البيانات (Validation)

- **عزل الأنواع في الفئات**: يمكن تسجيل نفس الاسم (مثل "مكافأة") كـ "نوع إيداع" و "نوع مصروف" في نفس الوقت دون تعارض.
- **تحديث البيانات**: تدعم الطلبات تحديث السجلات دون الاعتراض على الأسماء الموجودة مسبقاً للسجل نفسه (`ignore` logic).

---

## 6. الواجهات البرمجية (API Endpoints)

### المعاملات المالية (Transactions)

| العملية     | المسار                               | نوع البيانات          | المرفقات             |
| :---------- | :----------------------------------- | :-------------------- | :------------------- |
| تسجيل إيداع | `POST /api/finance/deposits`         | `multipart/form-data` | مدعوم (`attachment`) |
| تسجيل مصروف | `POST /api/finance/expenses`         | `multipart/form-data` | مدعوم (`attachment`) |
| تحويل مالي  | `POST /api/finance/transfers`        | `multipart/form-data` | مدعوم (`attachment`) |
| عرض تفاصيل  | `GET /api/finance/transactions/{id}` | `JSON`                | يظهر المسار كاملاً   |

### طرق الدفع والفئات

- `GET /api/finance/payment-methods`: عرض طرق الدفع المتاحة.
- `GET /api/finance/categories/expenses`: عرض أنواع المصروفات.
- `GET /api/finance/categories/income`: عرض أنواع الإيداعات.

## 7. السجلات والتدقيق (Logging)

يتم تسجيل كافة العمليات المالية في سجلات النظام مع تفصيل المستخدم، الشركة، ونوع العملية لضمان قابلية التتبع (Audit Trail).
